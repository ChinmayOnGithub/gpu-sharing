# Implementing Fractional GPU Allocation 

Kubernetes natively only allows **integer GPU requests** (e.g. `nvidia.com/gpu: 1`).  Pods cannot request `0.25 GPU` or similar【43†L152-L156】. NVIDIA’s device plugin reflects this: GPUs are advertised as whole devices, so the Kubernetes scheduler will only assign an entire GPU to a pod【35†L282-L285】【43†L152-L156】. To meet your project’s goal (multiple pods sharing one RTX 3060 without MIG or simple time-slicing), you must build a **custom GPU sharing layer** on top of Kubernetes. 

- **GPU Feature Limits:** The RTX 3060 does *not* support MIG or virtualization. Without hardware partitioning, the GPU must be multiplexed by software. NVIDIA provides CUDA MPS (Multi-Process Service) for concurrent usage, but it only time-slices compute and shares memory equally among clients【57†L842-L851】. CUDA MPS can enforce equal memory quotas and limit each client’s compute, but it does **not** truly isolate or assign fixed GPU “fractions” in hardware【57†L889-L897】. In other words, Kubernetes will still see one device; MPS simply allows N processes to share that device’s time and memory (each MPS client gets, say, 1/10th of the GPU memory with 10 replicas)【57†L889-L897】.

- **Software Partitioning Research:** Recent research shows software-level GPU partitioning is feasible. For example, the *Guardian* system splits the GPU’s address space and instruments kernels to provide **memory isolation** between tenants【41†L59-L66】. It “divides the common GPU address space into separate partitions” and checks all GPU calls at the PTX level to ensure a process cannot access another’s memory【41†L59-L66】. This approach is transparent to applications and has only ~5–10% overhead【41†L59-L66】. While Guardian was built for data-center GPUs, the same idea can inspire your design: you could map each pod to a software “slice” of GPU memory.

- **Device Plugin Modifications:** You will need to modify or extend the NVIDIA device plugin so Kubernetes sees fractional resources. One approach is to **advertise “virtual GPU” slices** instead of raw GPUs. For instance, configure the plugin to expose 8 “GPU slices” on a node (advertising something like `nvidia.com/gpu-slice: 8`), so that four pods could each request `gpu-slice: 2` to get 25% of the GPU. The plugin’s code already supports MPS-based sharing: e.g. a snippet from the plugin documentation shows how setting `sharing.mps.replicas: 10` on 8 GPUs makes Kubernetes see 80 GPU units, each limited to 1/10th of memory and compute【57†L889-L897】. You can adapt this mechanism to your custom slices. For example, setting `"sharing": {"mps": {"resources": [{"name": "nvidia.com/gpu","replicas": 8}]}}` would tell the plugin to advertise 8× the usual capacity, with each advertised GPU representing 1/8th of the real GPU’s memory and compute【57†L889-L897】. 

- **GPU Resource Manager Daemon:** Create a daemon (as a DaemonSet) to track and enforce allocations. This manager uses **NVML** to monitor actual GPU usage (utilization, memory)【54†L15-L23】. For each pod it assigns a logical slice (e.g. 2GB of VRAM and a compute share). For example, it could reserve 2 GB of the 12 GB VRAM for Pod A, 2 GB for Pod B, etc. NVML’s APIs let you query each process’s GPU memory usage【54†L15-L23】, so your daemon can ensure no pod exceeds its quota. For compute control, you can either rely on CUDA MPS (which can limit each client’s active threads【57†L843-L851】) or implement a simple fair-scheduling scheme yourself (e.g. using CUDA stream priorities or checking clock stats via NVML). 

- **Kubernetes Integration:** Modify your device plugin so that when a pod is scheduled, it registers with your GPU manager. For example, on `Allocate()` the plugin can call your daemon (via gRPC) to obtain a slice ID and memory quota. The pod’s container can be annotated with that ID. Then, inside the container’s startup (using an entrypoint script or a lightweight proxy), intercept CUDA initialization to attach it to the given slice (e.g. set up its CUDA context in MPS or pass a memory limit environment). Alternatively, use **RuntimeClass** or mount a special volume so each container only sees its slice. The key is: the device plugin only needs to create the illusion of multiple devices; the real enforcement happens in your daemon. 

- **Experiment Plan:** On RTX 3060 hardware, test with a small CUDA workload (e.g. a vector-add loop or PyTorch jobs). Use your daemon to allocate (say) 2 GB VRAM slices to each of 6 pods. Measure (via NVML) that no pod’s usage exceeds its 2 GB quota. Also check overall GPU utilization: if you launch workloads that individually only use ~20% of the GPU, your scheme should allow multiple to run simultaneously without oversubscription beyond each quota. Compare this to the baseline (one pod at a time). Adjust quotas and scheduling until all slices maintain isolation.

**References:** NVIDIA’s documentation explains that GPUs are normally advertised as integer resources【35†L282-L285】【43†L152-L156】, and shows how time-slicing/MPS can expose more “virtual GPUs” with fixed memory fractions【57†L889-L897】. Research like *Guardian* demonstrates that dividing GPU memory into partitions and trapping out-of-bounds accesses is feasible【41†L59-L66】. NVML provides APIs for monitoring GPU and per-process memory usage【54†L15-L23】. Using these tools and ideas, your project can implement a custom device plugin and GPU manager that together allocate fractional GPU slices to pods without relying on MIG or built-in time slicing.
